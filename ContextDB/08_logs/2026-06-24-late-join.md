# 2026-06-24 — Late-join standup delivery

Phase 2 follow-on. A member added (or granted **Report**) after today's run already opened now
gets today's standup instead of waiting for the next scheduled day. Spec:
[2026-06-24-late-join-standup-design.md](../../docs/superpowers/specs/2026-06-24-late-join-standup-design.md).
Plan: [2026-06-24-late-join-standup.md](../../docs/superpowers/plans/2026-06-24-late-join-standup.md).

## The gap

`openRun` fans out `send-standup-dm` to `can_report` members **at the moment the run opens** (the
scheduled time). A member added *after* that wasn't in the fan-out, so they got nothing that day.

## What shipped

- **`SendDmJob` + `SEND_DM_JOB` → `@poddaily/shared`** (`queue-contract.ts`). The worker re-exports
  the type and uses the const; this lets the **web** enqueue a `send-dm` job without importing
  `apps/worker` (excluded from the web image). Behavior-preserving for the worker.
- **Web queue access** — `bullmq` is now a `web` dependency; `apps/web/lib/queue.ts` is a
  lazily-constructed `Queue(QUEUE_NAME)` singleton (HMR-safe via `globalThis`, mirroring `lib/db.ts`)
  + `enqueueSendDm(job)` with the same job options as the worker's `makeEnqueueSend`.
- **`enqueueLateJoinIfOpen(memberId, enqueue?)`** (`apps/web/lib/late-join.ts`) — guards, short-
  circuiting in order: member is a reporter (`can_report`); the team's standup exists and is active
  (`is_active !== false`); a run exists for **today** (`scheduled_date = anchorDate(tz, now)`, any
  status); the member has **no report row** for that run yet → enqueue
  `send-dm { runId, standupId, slackUserId, slackDisplayName }`. The worker's `send-dm` then posts
  intro + Q1, inserts the report, and enqueues the 4h timeout (idempotent on the report row).
- **Both call sites** — the team detail page's **add-member** and **set-permissions** server
  actions call `enqueueLateJoinIfOpen` (best-effort: try/catch, never blocks the action). Calling
  it on any permission change is safe — the `can_report` + no-existing-report guards filter out
  view/edit toggles and already-reported members.

## Notable decisions

- **Enqueue-on-change, not a sweeper** — immediate, reuses the tested `send-dm` path (incl. the
  timeout). A periodic sweeper would add latency + scheduling for a case the onboarding hook covers.
- **Must enqueue (not direct web send)** — `send-dm` also enqueues the per-report 4h timeout; a
  direct web DM would leave the report with no timeout, hanging the run.
- **Gate on run-open-today** — the precise gap. No wall-clock window needed (runs aren't open at
  midnight). Finalized runs still count (no status filter) — late reports append to today's thread.
- **`web` gains Redis** — deploy note: set `REDIS_URL` on the `web` service. Without it the member
  is still added; only the same-day catch-up DM is skipped (logged, swallowed).
